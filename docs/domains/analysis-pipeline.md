# 분석 파이프라인

## 개요

분석 파이프라인은 웹 페이지의 텍스트를 분석하여 팩트체크 결과를 생성하는 일련의 처리 과정입니다. 이 문서는 파이프라인의 각 단계, 데이터 흐름, 에러 처리, 재시도 전략을 정의합니다.

## 단계별 상세 흐름

### Phase 1: 분석 요청

```
[사용자] → "분석 시작" 버튼 클릭
    │
    ▼
[Popup] → chrome.runtime.sendMessage({ type: 'REQUEST_ANALYSIS' })
    │
    ▼
[Background] → 메시지 수신, 처리 시작
```

**처리 내용:**
1. Popup에서 "분석 시작" 버튼 클릭
2. `REQUEST_ANALYSIS` 메시지를 Background로 전송
3. Background에서 현재 활성 탭의 URL과 제목 조회

**에러 처리:**
- 활성 탭 없음 → "분석할 페이지가 없습니다" 에러 반환
- 탭 접근 불가 → "페이지에 접근할 수 없습니다" 에러 반환

### Phase 2: 텍스트 추출

```
[Background] → content script에 메시지 전송
    │
    ▼
[Content Script] → readability로 본문 추출
    │
    ▼
[Background] → 추출된 텍스트 수신
```

**처리 내용:**
1. Background에서 Content Script에 `EXTRACT_PAGE_CONTENT` 메시지 전송
2. Content Script에서 `@mozilla/readability`로 본문 영역 탐지
3. 추출된 텍스트, 제목, URL을 Background로 반환

**텍스트 추출 알고리즘:**
1. 페이지 DOM에서 `<article>`, `<main>`, `<section>` 등 본문 영역 탐지
2. Readability 알고리즘으로 본문 점수 계산
3. 가장 높은 점수를 받은 영역의 텍스트 추출
4. 불필요한 태그 (스크립트, 스타일, 광고 등) 제거
5. 텍스트 정리 (공백 정리, 특수 문자 처리)

**에러 처리:**
- 본문 추출 실패 → 페이지 전체 텍스트로 폴백
- 텍스트 너무 짧음 (< 100자) → "분석할 텍스트가 부족합니다" 에러 반환
- DOM 접근 불가 → "페이지 구조를 분석할 수 없습니다" 에러 반환

### Phase 3: 캐시 확인

```
[Background] → 텍스트 해시 계산
    │
    ▼
[Cache Manager] → 캐시 조회
    │
    ├─── 캐시 히트 → 기존 결과 반환
    │
    └─── 캐시 미스 → API 호출继续
```

**처리 내용:**
1. 추출된 텍스트의 SHA-256 해시 계산
2. `chrome.storage.local`에서 해당 해시의 캐시 조회
3. 캐시 유효 기간 (기본 7일) 확인
4. 유효한 캐시가 있으면 기존 결과 반환
5. 캐시가 없거나 만료된 경우 API 호출继续

**캐시 키 형식:**
```
factcheck_{textHash}
```

**에러 처리:**
- 캐시 읽기 실패 → 캐시 미스로 처리, API 호출继续
- 캐시 저장 실패 → 조용히 무시, API 호출继续

### Phase 4: LLM API 호출

```
[Background] → Provider 팩토리에서 Provider 선택
    │
    ▼
[Provider] → API 호출
    │
    ▼
[LLM API] → 응답 반환
    │
    ▼
[Provider] → 응답 파싱
```

**처리 내용:**
1. 설정에서 선택된 Provider에 따라 인스턴스 생성
2. `AnalysisRequest` 객체 구성
3. Provider의 `analyze()` 메서드 호출
4. API 응답을 `AnalysisResponse`로 파싱

**프롬프트 설계:**
```
당신은 팩트체크 전문가입니다. 다음 텍스트에서 주요 주장을 추출하고 각 주장의 진위를 판단해주세요.

텍스트:
{extracted_text}

다음 형식으로 응답해주세요:
{
  "claims": [
    {
      "text": "추출된 주장",
      "verdict": "true|false|mixed|unverifiable",
      "confidence": 0.0-1.0,
      "reasoning": "판정 근거",
      "sources": [
        {
          "title": "출처 제목",
          "url": "출처 URL (선택사항)",
          "reliability": 0.0-1.0,
          "type": "academic|government|news|reference|other"
        }
      ]
    }
  ],
  "overallScore": 0-100,
  "summary": "전체 요약"
}
```

**에러 처리:**
- API 키 없음 → "API 키를 설정해주세요" 에러 반환
- API 호출 실패 → 재시도 로직 실행 (최대 3회)
- 응답 파싱 실패 → "분석 결과를 해석할 수 없습니다" 에러 반환
-Rate limit → 대기 후 재시도

### Phase 5: 결과 처리

```
[Background] → API 응답 파싱
    │
    ▼
[Background] → AnalysisResult 객체 생성
    │
    ▼
[Cache Manager] → 결과 저장
    │
    ▼
[Background] → Popup에 결과 전송
```

**처리 내용:**
1. API 응답을 `AnalysisResponse`로 파싱
2. `AnalysisResult` 객체 생성 (ID, 타임스탬프 등 추가)
3. 캐시에 결과 저장
4. Popup에 `ANALYSIS_COMPLETE` 메시지 전송

**에러 처리:**
- 파싱 실패 → "분석 결과를 처리할 수 없습니다" 에러 반환
- 캐시 저장 실패 → 조용히 무시, 결과는 정상 반환

### Phase 6: 결과 표시

```
[Popup] → 결과 수신
    │
    ▼
[Popup] → result-view 렌더링
    │
    ▼
[사용자] → 결과 확인
```

**처리 내용:**
1. Popup에서 `ANALYSIS_COMPLETE` 메시지 수신
2. `result-view.ts`로 결과 렌더링
3. 전체 신뢰도 점수, 주장별 분석, 요약 표시
4. "내보내기" 버튼 활성화

## 재시도 전략

### API 호출 재시도

```typescript
interface RetryConfig {
  maxRetries: number        // 최대 재시도 횟수 (기본: 3)
  baseDelay: number         // 기본 대기 시간 (밀리초, 기본: 1000)
  maxDelay: number          // 최대 대기 시간 (밀리초, 기본: 10000)
  backoffMultiplier: number // 대기 시간 증가 배율 (기본: 2)
}
```

**지수 백오프 알고리즘:**
```
재시도 1: 1초 대기
재시도 2: 2초 대기
재시도 3: 4초 대기
재시도 4: 8초 대기
...
최대: 10초 대기
```

**재시도 대상 에러:**
- 네트워크 에러 (timeout, connection refused)
- Rate limit 에러 (429)
- 서버 에러 (500, 502, 503)

**재시도하지 않는 에러:**
- API 키 오류 (401, 403)
- 요청 형식 오류 (400)
- 리소스 없음 (404)

### 캐시 재시도

캐시 읽기/쓰기 실패 시 재시도하지 않음. 조용히 무시하고 API 호출继续.

## 엣지 케이스 처리

### 1. 빈 텍스트

**조건:** 추출된 텍스트가 100자 미만
**처리:** "분석할 텍스트가 부족합니다" 에러 반환
**원인:** 광고만 있는 페이지, 동영상 페이지, 빈 페이지

### 2. 너무 긴 텍스트

**조건:** 추출된 텍스트가 10,000자 초과
**처리:** 텍스트를 10,000자로 잘라서 분석
**원인:** 백과사전 항목, 긴 뉴스 기사

### 3. 비영어 텍스트

**조건:** 텍스트가 설정된 언어가 아닌 경우
**처리:** 설정된 언어로 프롬프트 작성, LLM에 번역 요청 포함
**원인:** 외국어 기사, 다국어 페이지

### 4. 기술적 텍스트

**조건:** 코드, 수식 등 기술적 내용이 많은 경우
**처리:** 일반 텍스트로 분석 시도, "unverifiable" 판정 가능성 높음
**원인:** 개발자 블로그, 기술 문서

### 5. 주관적 표현

**조건:** "최고", "가장 좋다" 등 주관적 표현이 많은 경우
**처리:** "unverifiable" 판정, 신뢰도 점수 낮게 설정
**원인:** 리뷰 기사, 광고, 의견 기사

### 6. 인용문이 많은 텍스트

**조건:** 다른 사람의 말을 인용한 내용이 많은 경우
**처리:** 인용문과 원래 주장을 구분하여 분석
**원인:** 뉴스 기사, 인터뷰 기사

### 7. API 응답 형식 불일치

**조건:** LLM이 지정된 형식대로 응답하지 않은 경우
**처리:** 재시도 (프롬프트 재전송), 최대 3회
**원인:** 모델 변경, 프롬프트 엔지니어링 필요

### 8. 부분적 응답

**조건:** LLM이 응답을 중간에 자른 경우
**처리:** 완전한 JSON 파싱 실패 시 재시도
**원인:** 토큰 제한 초과

## 성능 고려사항

### 응답 시간

- **캐시 히트:** < 100ms
- **텍스트 추출:** < 500ms
- **API 호출:** 2-10초 (텍스트 길이, 모델에 따라 다름)
- **전체 파이프라인:** 3-15초

### 메모리 사용

- Content Script: < 10MB
- Background Worker: < 50MB
- Popup: < 20MB

### 저장 공간

- 캐시 1건당: < 100KB
- 히스토리 1건당: < 200KB
- 최대 캐시 용량: 5MB (chrome.storage.local 제한 고려)

## 로깅

### 로그 레벨

| 레벨 | 용도 | 예시 |
|------|------|------|
| `INFO` | 일반적인 진행 상황 | "분석 시작", "텍스트 추출 완료" |
| `WARN` | 경고 상황 | "캐시 만료", "텍스트 길이 초과" |
| `ERROR` | 에러 상황 | "API 호출 실패", "텍스트 추출 실패" |
| `DEBUG` | 디버깅용 상세 정보 | "해시 계산", "API 요청 상세" |

### 로그 형식

```
[Level] [Component] [Timestamp] Message
```

**예시:**
```
[INFO] [FactChecker] [2024-01-15T10:30:00Z] 분석 시작: https://example.com
[DEBUG] [CacheManager] [2024-01-15T10:30:00Z] 캐시 조회: hash=abc123
[INFO] [FactChecker] [2024-01-15T10:30:02Z] API 호출 완료: provider=openai, 2.3초
[WARN] [TextExtractor] [2024-01-15T10:30:00Z] 텍스트 길이 초과: 15000자 → 10000자로 잘라냄
```

# AGENTS.md

## Project

Chrome Extension (Manifest V3) for web page fact-checking. Scaffolded with Vite + `@crxjs/vite-plugin`.

## Commands

```bash
npm run dev       # Vite dev server with HMR — loads extension into browser via chrome://extensions
npm run build     # Outputs to dist/
npm run preview   # Preview built extension
```

No lint, typecheck, or test commands exist yet. TypeScript checking is implicit via Vite — `tsc` can be run manually (`npx tsc --noEmit`) but is not scripted.

## Architecture

Three extension entrypoints defined in `manifest.json`:

| Entry      | File                                         | Role                                              |
| ---------- | -------------------------------------------- | ------------------------------------------------- |
| Popup      | `src/popup/index.html` + `src/popup/main.ts` | Browser action popup UI                           |
| Background | `src/background.ts`                          | Service worker (`"type": "module"`)               |
| Content    | `src/content.ts`                             | Injected into all pages (`run_at: document_idle`) |

Build is driven by `vite.config.ts` passing `manifest.json` to `@crxjs/vite-plugin`. Do not modify the manifest directly for build config — change `vite.config.ts` instead.

## Quirks

- `@crxjs/vite-plugin` bundles the extension — Chrome reloads the extension on file changes during `npm run dev`, but the service worker may cache. Use `chrome://extensions` → reload if behavior seems stale.
- `tsconfig.json` uses `verbatimModuleSyntax` — all imports must use `import type` for type-only imports.
- `tsconfig.json` has `noUnusedLocals` and `noUnusedParameters` enabled — unused code will fail type checking.
- `src/` is the only compiled directory (`include: ["src"]` in tsconfig).
- Output goes to `dist/` (gitignored). The built manifest is auto-generated by the crxjs plugin, not the source `manifest.json`.
- Permissions: `activeTab`, `scripting`, `host_permissions: <all_urls>` — already permissive, be intentional about adding more.

## Workflow

All work is divided into phases. Follow these rules:

- Before starting, analyse requirements and split the work into phases. Present each phase's goal, scope, and deliverables before beginning.
- Execute one phase at a time. Do not proceed to the next phase until the current one is verified and approved.
- Each phase must be independently verifiable (e.g. Phase 1 project scaffold / Phase 2 manifest & entry points / Phase 3 content script / Phase 4 popup UI / Phase 5 background integration / Phase 6 end-to-end verification).
- If phase ordering or scope is unclear, ask before proceeding — never guess.

## Rules

- Read files before modifying them.
- Never rename or reorder fields in `manifest.json` — `@crxjs/vite-plugin` depends on its structure.
- Follow existing code style: Prettier, ESM, ES2023, TypeScript strict options. Do not introduce custom rules.
- Before adding a new dependency, explain the rationale and alternatives, then wait for approval.
- Do not write code based on assumptions. Read the relevant files or ask questions first.
- Stay within the current phase's scope — do not touch files outside it.
- At the end of each phase, verify that `npx tsc --noEmit` and `npm run build` pass.
- UI text and user-facing copy must be written in Korean.
- Do not create git commits without explicit user approval.
- Even after a phase completes, commits are never made automatically — only on explicit request.
- Each phase should result in exactly one logical commit. Do not split one phase into multiple commits or merge multiple phases into one, unless the user explicitly requests otherwise.

## Reporting

- All reports are written in Korean.
- At the end of each phase, report using the format below:

  - **진행한 Phase:** 번호와 이름
  - **수행한 작업:** 변경/생성/삭제한 파일과 핵심 내용
  - **검증 결과:** 타입 체크·빌드·테스트 결과
  - **다음 Phase 계획:** 예정 작업 요약
  - **확인이 필요한 사항:** 질문·리스크 (없으면 "없음")

- Keep reports concise and factual. Omit unnecessary decoration.
- If errors occur, report the cause and proposed fix — do not hide failures.
